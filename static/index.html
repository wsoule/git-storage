Key findings
The numbers are not subtle.
580×
Exists — BadgerDB vs MinIO/S3
Git calls Exists constantly during push to avoid resending objects. BadgerDB handles 557k ops/sec. MinIO handles 951 — every check is an HTTP round trip regardless of object size.
4.9×
Concurrent Put — BadgerDB vs SQLite (1KB)
Without WAL mode and a connection limit of 1, SQLite produces SQLITE_BUSY errors under concurrent load. With those fixes, BadgerDB still wins by nearly 5× at small object sizes.
≈1ms
MinIO floor — per request, any size
A 1KB Get on MinIO costs roughly the same as a 1MB Get. The HTTP round trip is the bottleneck, not the data. Git's workload is small objects. This is a bad fit.
~1×
Large object Put — all backends converge
At 1MB, SQLite (49), BadgerDB (52), and MinIO (47) are within noise of each other. The bottleneck becomes I/O, not the storage layer. Backend choice matters most at small sizes.
